1. What is a task?

A task is the unit of work you are doing right now — a feature, a bugfix, an incident, a refactor. Instead of keeping your branch, chat conversation, test runs, CI status and pull request in separate tools, Polyglot AI ties them all to a single task. Every panel (Git, Tests, CI, Chat, Review) can re-scope itself to the active task, and writes its activity back onto a shared timeline.

The end result: when you come back to a task after lunch (or next Monday), you can open the Today panel, click the task, and immediately see the branch you were on, the last test run, any PR you opened, and the conversation you were having with the AI — without hunting for context.

Mental model. Think of a task as a folder that collects everything relevant to one piece of work. The folder is invisible — it just makes the panels smarter.

2. Core concepts

Kind

The flavour of work. Each kind has its own colour in the sidebar.

KindWhen to use
feature New functionality. FEATURE is the default choice when creating a task.
bugfix Short loop focused on reproducing then fixing a specific defect.
incident Production issue. Auto-created by the CI panel when a run fails on your branch.
refactor Restructuring with no new behaviour.
explore Data or codebase exploration (scratch work that may not ship).
chore Dependency bumps, config, infra, maintenance.

State

Where the task sits in its lifecycle. Used to group the sidebar and drive colours.

StateMeaning
PLANNING You’re defining the task. New tasks start here. AI may draft a plan checklist.
ACTIVE You’re working on it. This is where most of a task’s life is spent.
REVIEW PR is open, waiting on review / CI. Moving to REVIEW happens automatically when you open a PR.
BLOCKED Waiting on something external. Use this to park a task without losing its context.
DONE Merged or closed. Still visible in the sidebar under "DONE (recent)".
ARCHIVED Hidden from normal views. Use archive to hide older finished work.

Active task

Exactly one task is active at any moment (or none). The active task is highlighted in the Tasks sidebar with a blue border, and every task-aware panel re-scopes to it:

  • Chat switches to the task’s saved conversation (or binds the next new one to it).
  • Git records commits, branch creation, and PRs onto the task.
  • Tests saves a pass/fail snapshot after each run.
  • CI/CD records the latest status of the task’s branch.
  • Review defaults to branch vs main and logs each review pass.

3. The UI

Today panel

The landing page that answers “what should I do right now?”. Open it from the activity bar (first icon) or with Ctrl+Shift+H. It shows three sections:

  1. Active Tasks — cards for any ACTIVE / PLANNING / REVIEW task. Click a card to make it the active task.
  2. Attention — failed CI runs and open PRs on your branches, fetched via the gh CLI when available. Refreshes on project open and every 60 seconds.
  3. Quick Actions — buttons for common commands (New task, Run all tests, Refresh CI, etc.).

Tasks sidebar panel

Open from the activity bar (second icon) or with Ctrl+Shift+J. Shows every non-archived task in the current project, grouped by state in this order: ACTIVE → PLANNING → REVIEW → BLOCKED → DONE (recent).

Task card

Each card shows:

  • A coloured dot for the kind.
  • The task title.
  • A meta line: kind · ⎇ branch · tests passed/total · CI status · age (e.g. 3m ago).

Interactions

ActionEffect
ClickMake the task active.
Double-clickOpen the full Task Detail dialog.
Right-clickContext menu: activate, open details, move to any state, archive, delete.
+ buttonOpen the New Task dialog.
Refresh buttonReload from disk.
Expand buttonOpen Tasks in a separate, larger window that stays in sync with the sidebar.

New Task dialog

Pick a kind, type a title (required), and an optional description. The description is the strongest signal for the AI — it is injected into the system prompt when you chat while the task is active.

Task Detail dialog

Opened by double-clicking a card or by choosing Open details… from the context menu. Contents top-to-bottom:

  1. Header — kind dot, title, and a meta line (kind · state · branch · PR).
  2. Description — read-only text block (shown only if set).
  3. Plan — read-only checklist if the task has plan steps. Glyphs: ☐ pending, ◐ in progress, ☑ done, ⊘ skipped.
  4. Stats card — test ratio, CI symbol, files-touched count, PR number (only the relevant ones render).
  5. Timeline — newest first, one row per note.
  6. Actions bar:
    • Open PR — if a PR URL is set.
    • Copy as standup — builds a markdown summary (title, branch, PR, recent events, tests, CI) and copies it to the clipboard. Paste straight into your daily standup.
    • State transition buttons (Start work / Mark ready for review / Mark done) for anything that isn’t already the current state.
    • Close.
Why the dialog closes on state change. Rebuilding the action row in place was leaving orphaned widgets visible, so the dialog closes after a state transition. The sidebar and Today page refresh instantly, and you can reopen the dialog to see the new timeline entry.

4. Task lifecycle — step by step

A typical feature flow, end to end:

  1. Open the project. The task manager auto-activates the most recently updated task that is still in ACTIVE / PLANNING / REVIEW. If none, active is cleared.
  2. Create the task. Press Ctrl+Shift+J, click +, pick feature, type a title like “Add CSV export to user reports”, fill in a short description. It’s created in PLANNING and becomes active.
  3. Chat. Open the chat panel. Because the task is active, the context builder injects a task block into the system prompt (“You are helping with task: Add CSV export…”). Start a new conversation — when it first persists, its ID is written to chat_session_id on the task, so next time you activate the task the chat switches to that conversation automatically.
  4. Create a branch. In the Git panel, create a branch (e.g. feat/csv-export). The task records branch, base_branch, and a branch_created note.
  5. Move to ACTIVE. From the Task Detail dialog, click Start work. (Or via the sidebar right-click menu.)
  6. Work & commit. Each commit adds a committed note and appends the touched files to modified_files. If the task didn’t yet have a branch, the first commit fills it in.
  7. Run tests. In the Tests panel, run your pytest suite. When it finishes, a TestRunSnapshot is saved onto the task and a tested note appears in the timeline. The sidebar card now shows 12/13 tests.
  8. AI code review. Open the Review panel. With a task branch set, it defaults to branch vs main. Run a review — results are logged as review_clean / review_findings / review_failed.
  9. Open a PR. From the Git panel, click Generate PR description and push. The task moves to REVIEW, pr_url and pr_number are saved, and a pr_opened note is added.
  10. Watch CI. The CI/CD panel filters runs by the task’s branch. Each status change writes a last_ci_run snapshot and a ci_run note. Today’s Attention section surfaces the failure if it goes red.
  11. Merge & mark done. After the PR merges, open details, click Mark done. The task drops into the DONE group.
  12. Archive (optional). Once the task is noise, right-click → Archive to hide it from normal views. Data is kept.

5. What each panel records automatically

You never have to touch the task manager directly. These side-effects happen whenever a task is active:

Git panel

  • Branch create → if the task has no branch, fills branch + base_branch, adds a branch_created note.
  • Commit → adds a committed note with sha + message, merges the staged files into modified_files.
  • Push → adds a pushed note.
  • PR opened (via the PR summary dialog) → saves pr_url, pr_number, moves state to REVIEW, adds a pr_opened note.
  • Branch switch → cards that reference a branch use the task’s branch for comparisons (e.g. diff vs base).

Tests panel

  • After a pytest run, writes last_test_run = TestRunSnapshot(passed, failed, skipped, timestamp).
  • Adds a tested note summarising the result.

CI/CD panel

  • Filters the runs table to the task’s branch when one is set.
  • Records last_ci_run = CIRunSnapshot(status, workflow, url, timestamp) on every status change.
  • When a run fails and there is no active task, you can Import failure as incident — a new INCIDENT task is created with the workflow details and made active.

Chat panel

  • Subscribes to task:changed. When a task with a chat_session_id becomes active, the panel loads that conversation.
  • Hands the task object to the context builder so the system prompt includes a “You are helping with task X” block.
  • On the first persist of a new conversation, writes chat_session_id back onto the task.

Review panel

  • Defaults the review mode to branch vs main when the active task has a branch.
  • Records each review run on the timeline as review_clean, review_findings, or review_failed depending on the outcome.

6. Manual actions you can take

Create a task

  • Tasks sidebar → + button.
  • Today panel → New task quick action.
  • CI/CD panel → Import failure as incident (creates an INCIDENT task).

Switch the active task

  • Single-click any card in the Tasks sidebar or Today panel.
  • Right-click a card → Make active.

Change state

  • Task Detail dialog → Start work / Mark ready for review / Mark done buttons.
  • Sidebar context menu → Move to: … for any state.

Archive or delete

  • Context menu → Archive (reversible, keeps data) or Delete (permanent).

Copy a standup update

Task Detail dialog → Copy as standup. The copied markdown looks like:

**Add CSV export to user reports** (feature)
Branch: `feat/csv-export`
PR: https://github.com/you/repo/pull/42

Recent:
- 14:02  Commit: stream rows to tempfile
- 13:50  Tests: 12/13 passing
- 13:10  PR opened

Tests: 12/13 passing
CI: success

Open the PR

Task Detail dialog → Open PR (only if pr_url is set).

8. Where data lives & how events flow

Storage

Tasks are stored in their own SQLite database, independent of the main app DB:

~/.config/polyglot-ai/tasks.sqlite

One tasks table with JSON-blob columns for nested structures (plan, notes, modified_files, last_test_run, last_ci_run). Indexed by (project_root, state) and (project_root, updated_at DESC) so the sidebar query is cheap.

Backup this file if you care about the history of what you’ve been doing. Deleting it is a factory reset for the task system and cannot be undone.

Components

ModuleRole
core/tasks.pyDataclasses: Task, TaskKind, TaskState, PlanStep, TaskNote, TestRunSnapshot, CIRunSnapshot.
core/task_store.pySynchronous SQLite CRUD. save(), get(), list_tasks(), delete().
core/task_manager.pySingleton that owns the active task and emits events. Initialised once in app.py via init_task_manager(event_bus).
ui/panels/tasks_panel.pySidebar list and new-task dialog.
ui/panels/today_panel.pyLanding page with active tasks + attention + quick actions.
ui/dialogs/task_detail_dialog.pyFull-screen detail viewer.

Event bus topics

All panels subscribe to these through the shared EventBus:

TopicPayloadEmitted by
task:changedtask (or None)active task changed / created / cleared
task:list_changed—create, delete, archive, project switch
task:state_changedtask, old_state, new_stateupdate_state()
task:note_addedtask, noteadd_note()

Wiring order (important)

In app.py, init_task_manager(event_bus) runs before any panel’s set_event_bus() is called. Panels (git, tests, CI, chat…) fetch the singleton via get_task_manager() during wiring. If the order was reversed, those panels would grab a manager without an event bus and every write would silently fail to propagate.

Project switching

ui_wiring.wire_project_events listens for project:opened and calls task_manager.set_project_root(path). That re-scopes the sidebar and auto-activates the most recent ACTIVE / PLANNING / REVIEW task in the new project, or clears the active task if there are none.

9. Recipes

Quick bugfix loop

  1. Ctrl+Shift+J, +, pick bugfix, title “NPE in invoice totals”. Description: paste the traceback.
  2. Ask the chat to explain / fix. Apply the patch.
  3. Run the targeted test in the Tests panel.
  4. Commit, push, open PR — state auto-advances to REVIEW.
  5. Task Detail → Copy as standup for your daily update.

Triage a CI failure

  1. Open Today panel. The failed run shows up under Attention.
  2. Click Import failure as incident in the CI panel row, or the equivalent action in the attention card.
  3. An INCIDENT task is created and activated. The CI snapshot and a ci_failure_imported note are seeded so you have the workflow and URL in hand.
  4. Proceed as for a bugfix.

Parallel tasks

You can have as many PLANNING / ACTIVE tasks as you like — only one is active. Click a different card in the Tasks sidebar to hot-swap. Chat switches to the other task’s conversation automatically; git actions will now attach to the new task. Nothing is lost on the old one.

Park a task

Waiting on a code owner? Move the task to BLOCKED via the context menu or detail dialog. It stays in the sidebar (in the BLOCKED group) so you don’t forget it, but it’s out of your ACTIVE list.

10. Troubleshooting

“No project open” in the Tasks sidebar

The task manager is scoped per project. Open a folder via File → Open Project first; the sidebar will re-populate.

My commit didn’t show up on the timeline

Check the following:

  • Was a task active at the moment you committed? The Git panel is a no-op on the task side when there is no active task.
  • Was the commit made through the Git panel, not a terminal? External commits are not detected by the panel.

Chat didn’t switch when I clicked the task

  • If the task has never been bound to a conversation (chat_session_id is empty), the chat panel will leave the current conversation alone and bind the next new one.
  • If you’re already on the right conversation, nothing happens (guarded against a reload race with _persist_conversation).

The state transition buttons closed my dialog

Expected — see the callout in section 3. Reopen by double-clicking the card to see the new timeline entry.

I deleted a task by mistake

Delete is permanent. If you want a safer option, prefer Archive — it hides the task but keeps all its data. If you need to recover a deleted task, restore ~/.config/polyglot-ai/tasks.sqlite from backup.

11. FAQ

Can I run tasks across multiple projects?

Tasks are scoped to a project root. Opening a different project switches the sidebar to that project’s tasks. Archived tasks remain attached to their original project.

Is the plan checklist filled in by AI?

The model and UI support it (read-only for now). A FEATURE task can carry plan steps in task.plan; the Detail dialog renders them with status glyphs. An AI plan generator is the next thing on the workflow roadmap; when it lands it will populate those steps automatically during PLANNING.

Where are the tests for this feature?

tests/core/test_tasks.py covers the model, store (round-trip, filtering, project scoping, delete, corrupt-JSON fallback), and manager (CRUD, state transitions, event emission, whitelist guard, listener-exception isolation, project switching). The UI panels themselves are not tested as there is no Qt test harness in the repo.

How do I reset everything?

Close the app and delete ~/.config/polyglot-ai/tasks.sqlite. The next launch will recreate an empty schema. This does not affect your chat history, settings, or any other data — tasks live in their own database.

Can I script against the task store?

Yes — TaskStore is a plain Python class with no async. From a script: 

from polyglot_ai.core.task_store import TaskStore
from polyglot_ai.core.tasks import TaskState

store = TaskStore()
for t in store.list_tasks("/path/to/project", state_filter=[TaskState.ACTIVE]):
    print(t.title, t.branch, t.pr_url)

The schema is stable; feel free to build custom reports on top of it.
